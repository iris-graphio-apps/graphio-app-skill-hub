# 스킬 작성 방법론 (요약)

> Anthropic 공식 `skill-creator`(Apache-2.0)의 방법론을 회사 hub 용으로 압축·재구성한 것이다. 무거운 정량 평가(subagent 벤치마크, eval-viewer, description 최적화 loop)가 필요하면 설치된 공식 skill-creator 를 직접 쓴다. 이 문서는 자립적으로 "좋은 스킬"을 쓰는 데 필요한 핵심만 담는다.

## 1. Intent 포착 (먼저 묻는다)

1. 이 스킬이 Claude 가 **무엇을** 할 수 있게 하나?
2. **언제** 트리거되나? (사용자 표현/맥락)
3. 기대 **출력 형식**은?
4. 테스트 케이스로 검증할 가치가 있나? (파일 변환·데이터 추출·코드 생성처럼 객관적으로 검증 가능한 출력이면 O, 글쓰기 스타일 같은 주관적 출력이면 보통 불필요)

대화에 이미 "이걸 스킬로 만들자"는 워크플로가 있으면, 사용한 도구·단계 순서·사용자 교정·입출력 형식을 히스토리에서 먼저 추출하고 사용자에게 빈칸만 확인한다.

## 2. 스킬 구조 (Anatomy)

```
skill-name/
├─ SKILL.md (필수)
│  ├─ YAML frontmatter (name, description 필수)
│  └─ Markdown 지침
└─ 번들 리소스 (선택)
   ├─ scripts/    - 결정적·반복 작업용 실행 코드
   ├─ references/ - 필요할 때 컨텍스트로 읽는 문서
   └─ assets/     - 출력에 쓰는 파일(템플릿·아이콘 등)
```

### Progressive disclosure (3단 로딩)

1. **메타데이터(name+description)** — 항상 컨텍스트에 (~100단어)
2. **SKILL.md 본문** — 트리거되면 로드 (이상적으로 <500줄)
3. **번들 리소스** — 필요할 때만 (무제한, 스크립트는 로드 없이 실행 가능)

핵심: SKILL.md 는 500줄 아래로. 길어지면 계층을 한 단 더 나누고, "다음에 어디를 읽어야 하는지" 명확한 포인터를 남긴다. 큰 reference(>300줄)는 목차를 넣는다. 여러 도메인/프레임워크를 다루면 variant 별로 `references/<variant>.md` 로 쪼개고 본문은 선택 로직만 둔다.

## 3. description = 트리거의 전부

description 이 스킬 호출 여부를 결정하는 1차 메커니즘이다. **무엇을 하는지 + 언제 쓰는지**를 모두 담고, "언제 안 쓰는지"(형제 스킬로 보내기)도 넣는다. Claude 는 스킬을 **과소트리거**하는 경향이 있으니 약간 pushy 하게 쓴다.

- 나쁨: `내부 데이터를 보여주는 대시보드 만드는 법.`
- 좋음: `내부 데이터를 보여주는 대시보드 만드는 법. 사용자가 대시보드·데이터 시각화·내부 지표를 언급하거나 회사 데이터를 어떤 형태로든 보여주려 하면, '대시보드'라고 명시하지 않아도 반드시 이 스킬을 쓴다.`

회사 hub 의 추가 하드 규칙: **30–1024자, 꺾쇠 `< >` 금지**(→ `lint_description.py`).

## 4. 본문 작성 패턴

- **명령형(imperative)** 으로 쓴다.
- **왜**를 설명한다. 무거운 대문자 MUST/NEVER 대신 이유를 주면, 똑똑한 모델이 rote 지침을 넘어 맥락에 맞게 판단한다. ALWAYS/NEVER 를 남발하고 있으면 yellow flag — 재구성해서 이유를 설명한다.
- 출력 형식이 고정이면 정확한 템플릿을 박는다.
- 예시(Input/Output)는 유용하다. 단 너무 특정 예시에 과적합되지 않게 일반적으로 쓴다.
- 초안을 쓰고 **새 눈으로 다시 읽어** 다듬는다.

### Principle of lack of surprise

스킬은 멀웨어·익스플로잇·기만적 의도를 담지 않는다. 설명대로의 의도여야 하고, 무단 접근·데이터 유출을 돕는 스킬은 만들지 않는다.

## 5. 가벼운 테스트 (핵심만)

초안 후 실제 사용자가 말할 법한 **테스트 프롬프트 2–3개**를 만들어 사용자와 합의하고 돌려본다. 회사 hub 흐름에서는 여기서 정량 벤치마크까지 가지 않아도 된다 — 스킬을 직접 적용해 결과를 보여주고 "이렇게 나오는 게 맞나?"를 사용자에게 묻는 정성 점검으로 충분하다.

반복 루프: 피드백 → 일반화해서 개선(과적합 금지) → 군더더기 제거 → 다시 테스트. 여러 테스트가 공통으로 같은 헬퍼 스크립트를 만들면, 그건 그 스크립트를 `scripts/` 에 번들하라는 신호다.

정량 벤치마크(subagent with-skill vs baseline)·eval-viewer·description 최적화 loop 가 필요하면 → 설치된 공식 `skill-creator` 사용.

## 6. 회사 hub 등록으로 마무리

좋은 스킬을 다 썼으면 [`registry-conventions.md`](registry-conventions.md) 의 흐름으로 등록한다: `lint_description.py` → `registry_entry.py --write` → `validate_registry.py`. 이게 끝나야 "done".
